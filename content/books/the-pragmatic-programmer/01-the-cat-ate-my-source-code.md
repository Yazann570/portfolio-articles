# The Cat Ate My Source Code

## One-line takeaway

A pragmatic programmer takes responsibility and provides options instead of excuses.

## Important ideas

- Take ownership of your work.
- Admit mistakes honestly.
- Do not blame tools, vendors, teammates, or circumstances.
- When there is a problem, explain what can be done next.
- Before reporting bad news, think about possible solutions.

## My understanding

This section reminds me that being a developer is not only about writing code. It is also about being responsible for the result. If something fails, I should not just explain why it failed. I should also help move the situation forward.

## Example

Instead of saying:

> I could not finish the feature because the API was confusing.

A better response is:

> I could not finish the feature because the API behavior was unclear. I can either create a small prototype first, ask for clarification, or implement the simpler version now and improve it later.

## How I can apply this

When I face a bug, delay, or unclear task, I should communicate early and bring possible solutions.

## Action item

Before I tell someone about a problem, I will prepare at least two possible options.
