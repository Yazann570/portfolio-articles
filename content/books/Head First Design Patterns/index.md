# Head First Design Patterns

These are my personal notes from reading *Head First Design Patterns*.

The goal of these notes is not to copy the book. The goal is to keep a clean portfolio-friendly summary of each pattern, when to use it, and what problem it solves.

## Core design ideas from the book

- Encapsulate what varies.
- Favor composition over inheritance.
- Program to interfaces, not implementations.
- Strive for loosely coupled designs between objects that interact.
- Classes should be open for extension but closed for modification.
- Depend on abstractions, not concrete classes.
- Talk only to your close friends.
- Do not call us, we will call you.
- A class should have only one reason to change.

## How I will use these notes

- Use the pattern name as a shared vocabulary when discussing design.
- Start with the problem first, not the pattern.
- Avoid forcing a pattern into code just because I learned it.
- Prefer the simplest design that keeps the code flexible for expected change.
